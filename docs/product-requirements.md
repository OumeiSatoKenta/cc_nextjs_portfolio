# プロダクト要求定義書 (Product Requirements Document)

## プロダクト概要

### 名称
**Portfolio Site** - エンジニアポートフォリオサイト

### プロダクトコンセプト
- **Static Exportによる高速配信**: Next.js Static Exportで生成した静的HTMLをCloudFront + S3で配信し、高速かつ低コストで運用する
- **コードで語るポートフォリオ**: プロジェクトの構成図・設計判断・使用技術をカード形式で紹介し、技術力をコードとアーキテクチャで示す
- **最小構成・最大効果**: 個人ポートフォリオとして必要十分な5ページ構成で、メンテナンスコストを最小限に保つ

### プロダクトビジョン
マルチクラウド対応力・AI駆動開発・パフォーマンス改善の3つの強みを、実際のプロジェクト成果とともに提示するエンジニアポートフォリオサイトを構築する。
サイト自体がIaC管理・CI/CD自動化・セキュリティ設計のショーケースとなり、「このサイトの構築プロセス自体が技術力の証明」となることを目指す。
静的サイトとしてシンプルに保ちながら、洗練されたデザインとアニメーションで訪問者に印象を残す。

### 目的
- 転職・副業・登壇時に自身の技術力・経歴・実績を効果的に伝える
- 公開リポジトリ・テックブログ記事・登壇資料への導線を一元化する
- サイト構築自体をポートフォリオ作品として活用する（Terraform IaC、GitHub Actions CI/CD、Next.js Static Export）

## ターゲットユーザー

### プライマリーペルソナ: 採用担当者・技術リード（30-45歳、IT企業）
- エンジニア候補の技術力を短時間で評価したい
- GitHubリポジトリやブログ記事を見て技術の深さを判断する
- ポートフォリオサイトは「第一印象」として数秒で判断する
- 経歴の一貫性と成長ストーリーを重視する
- 典型的なフロー: LinkedIn/Wantedly → ポートフォリオサイト → GitHub → 面談判断

### セカンダリーペルソナ: エンジニアコミュニティ参加者（25-40歳）
- connpass・X(Twitter)経由でプロフィールを確認する
- 登壇者やOSS貢献者のバックグラウンドに興味がある
- 技術ブログの著者情報として参照する

## 成功指標

### マイルストーン（リリース条件）
- サイト公開完了: 独自ドメインでアクセス可能な状態（2026年5月末）
- 全5ページの実装完了: トップ・About・Projects・Blog・Contact（2026年5月末）

### パフォーマンスKPI（リリース後継続計測）
- Lighthouse Performance スコア: 90以上（Static Exportのため達成見込み高）
- ページ読み込み時間(LCP): 2.5秒以内
- CLS (Cumulative Layout Shift): 0.1以下
- モバイル対応: レスポンシブデザインで主要画面サイズ（375px〜1440px）をカバー

## 機能要件

### コア機能(MVP)

#### F1: トップページ（/）

**ユーザーストーリー**:
採用担当者として、候補者の概要を数秒で把握するために、ヒーローセクションと強みのサマリーが欲しい

**受け入れ条件**:
- [ ] ヒーローセクションに名前・肩書き・キャッチコピーが表示される
- [ ] 3つの強みサマリー（マルチクラウド / AI駆動開発 / パフォーマンス改善）がカード形式で表示される
- [ ] 各セクション（About / Projects / Blog / Contact）への導線がある
- [ ] ファーストビューで訪問者が「何をしている人か」を理解できる

**優先度**: P0（必須）

#### F2: 経歴・スキルページ（/about）

**ユーザーストーリー**:
採用担当者として、候補者の経験とスキルセットを時系列で確認するために、経歴タイムラインとスキル一覧が欲しい

**受け入れ条件**:
- [ ] 経歴がタイムライン形式（Timeline コンポーネント）で表示される
- [ ] 各職歴に会社名・在籍期間・役割・主な成果が含まれる
- [ ] スキル一覧がカテゴリ別（言語、フレームワーク、クラウド、ツール等）で表示される
- [ ] 資格・学歴セクションがある

**優先度**: P0（必須）

#### F3: プロジェクト一覧ページ（/projects）

**ユーザーストーリー**:
採用担当者として、候補者の実装力と設計力を確認するために、公開プロジェクトの詳細と技術選定理由が欲しい

**受け入れ条件**:
- [ ] プロジェクトがカード形式（ProjectCard コンポーネント）で一覧表示される
- [ ] 各カードに、プロジェクト名・概要・使用技術タグ・GitHubリンクが含まれる
- [ ] 構成図や設計判断のハイライトが表示可能
- [ ] このポートフォリオサイト自体がプロジェクトとして掲載される

**優先度**: P0（必須）

#### F4: ブログリンク集ページ（/blog）

**ユーザーストーリー**:
訪問者として、候補者の技術的な発信内容を確認するために、テックブログ記事・Zenn記事へのリンク集が欲しい

**受け入れ条件**:
- [ ] 記事リンクがカード形式で一覧表示される
- [ ] 各カードに、記事タイトル・投稿日・プラットフォーム（Zenn等）・概要が含まれる
- [ ] 外部サイト（Zenn、その他ブログ）へのリンクが正しく開く（target="_blank"、rel="noopener noreferrer"）
- [ ] 記事データは `src/data/` 内のTSファイルで管理される

**優先度**: P0（必須）

#### F5: コンタクトページ（/contact）

**ユーザーストーリー**:
採用担当者として、候補者に連絡を取るために、各種SNS・プラットフォームへのリンクが欲しい

**受け入れ条件**:
- [ ] GitHub / X(Twitter) / connpass 等のリンクがアイコン付きで表示される
- [ ] リンクは新しいタブで開く
- [ ] メールアドレスへのリンク（`mailto:` 形式）またはメールアドレスの表示がある（問い合わせフォームのバックエンド処理はスコープ外）

**優先度**: P0（必須）

#### F6: 共通レイアウト

**ユーザーストーリー**:
訪問者として、サイト内をスムーズに回遊するために、一貫したヘッダー・フッターとナビゲーションが欲しい

**受け入れ条件**:
- [ ] 全ページ共通のHeader コンポーネント（ナビゲーションリンク付き）がある
- [ ] 全ページ共通のFooter コンポーネント（コピーライト・SNSリンク）がある
- [ ] 現在のページがナビゲーション上でハイライトされる
- [ ] モバイルではハンバーガーメニューまたは適切なモバイルナビゲーションが表示される
- [ ] App Router の `layout.tsx` で共通レイアウトを実装する

**優先度**: P0（必須）

#### F7: Static Export対応

**ユーザーストーリー**:
開発者として、S3 + CloudFrontで静的ホスティングするために、Next.js Static Exportが正しく動作する必要がある

**受け入れ条件**:
- [ ] `next.config.js` に `output: 'export'` が設定されている
- [ ] `trailingSlash: true` が設定されている（CloudFront + S3ルーティング対応）
- [ ] `images.unoptimized: true` が設定されている（Static Export制約）
- [ ] `npm run build` で `out/` ディレクトリに全ページが正しく出力される
- [ ] 各ページが `/about/index.html` 形式で出力される

**優先度**: P0（必須）

### 将来的な機能(Post-MVP)

> Post-MVP機能は実装着手時にユーザーストーリーと受け入れ条件を追記する。現時点ではスコープと優先度の記録を目的として列挙する。

#### F8: アニメーション・インタラクション

Framer Motionによるページ遷移アニメーション、スクロールアニメーション、ホバーエフェクトの追加。

**優先度**: P1（重要）

#### F9: ダークモード対応

Tailwind CSSの `dark:` バリアントを活用したダークモード切替機能。

**優先度**: P2（できれば）

#### F10: i18n（英語対応）

英語版ページの追加。海外企業へのアピール用。

**優先度**: P2（できれば）

#### F11: OGP画像の自動生成

各ページ用のOGP画像を動的に生成し、SNS共有時のプレビューを最適化。

**優先度**: P2（できれば）

## 非機能要件

### パフォーマンス
- Lighthouse Performance スコア: 90以上
- LCP (Largest Contentful Paint): 2.5秒以内
- INP (Interaction to Next Paint): 200ms以内（静的サイトのため自動達成の見込み）
- CLS (Cumulative Layout Shift): 0.1以下
- 静的ファイルサイズ: JavaScript バンドル合計 200KB以下（gzip後）

### ユーザビリティ
- レスポンシブデザイン: 375px（iPhone SE）〜 1440px（デスクトップ）まで対応
- アクセシビリティ: Lighthouse Accessibility スコア 90以上（Lighthouse CI または手動実行、デプロイ後計測）
- セマンティックHTML: 適切な見出し階層、landmark要素の使用

### 信頼性
- ビルドの再現性: `npm ci && npm run build` が任意の環境で成功する
- リンク切れゼロ: 外部リンクを含む全リンクが有効（デプロイ前の手動確認。外部サイトのURL変更は都度対応）

### セキュリティ
- 外部リンクに `rel="noopener noreferrer"` を付与
- 個人情報（電話番号・住所等）をソースコードに含めない
- 依存パッケージの既知の脆弱性がゼロ（`npm audit` でCI/CDパイプラインにて確認）

### SEO
- 各ページに適切な `<title>` と `<meta description>` を設定
- OGPタグ（og:title, og:description, og:image）を設定
- OGP画像: MVP時点は `public/images/og-image.png`（1200x630px）を静的ファイルとして配置。Post-MVPのF11で自動生成に移行予定
- `robots.txt` と `sitemap.xml` を出力

## デザインシステム

プロジェクトルートの `DESIGN.md` にVercel Inspiredデザインシステムを定義し、UI実装時の一貫性を担保する。

### デザイン方針
- **ライトモードベース**: 白背景 (`#ffffff`) + ニアブラック (`#171717`) テキスト
- **Geistフォントファミリー**: `next/font/google` で標準提供。Sans（見出し・本文）+ Mono（コード・技術ラベル）
- **Shadow-as-Border**: CSS borderの代わりに `box-shadow: 0px 0px 0px 1px rgba(0,0,0,0.08)` を使用
- **3ウェイト制**: 400（本文）/ 500（UI要素）/ 600（見出し）

### デザイントークンの適用
1. `DESIGN.md` — デザインの真実の源泉（AIエージェントが参照）
2. `tailwind.config.ts` — DESIGN.mdのトークンをTailwind CSSに反映（二重の制約レイヤー）

### カスタマイズ方針
Vercelテンプレートをベースに、ポートフォリオ固有のカスタマイズを適用:
- Workflow Accent Colors（Ship Red / Preview Pink / Develop Blue）はポートフォリオの強み3つ（マルチクラウド / AI駆動開発 / パフォーマンス改善）に対応させる
- ダークモード対応はPost-MVP (P2) で追加

## データ管理

コンテンツデータは `src/data/` 配下のTypeScriptファイルで管理する（CMS不使用）:

| ファイル | 内容 |
|---------|------|
| `src/data/skills.ts` | スキル一覧（名前、カテゴリ、習熟度） |
| `src/data/projects.ts` | プロジェクト一覧（名前、概要、技術タグ、リンク） |
| `src/data/career.ts` | 経歴データ（会社名、期間、役割、成果） |
| `src/data/education.ts` | 資格・学歴データ（種別、名称、機関、取得日） |
| `src/data/blog.ts` | ブログ記事リンク（タイトル、URL、日付、プラットフォーム） |
| `src/data/social.ts` | SNS・外部リンク（プラットフォーム名、URL、アイコン） |

## CI/CD

### デプロイフロー（GitHub Actions）

```
main push → npm ci → npm run build → aws s3 sync out/ → CloudFront invalidation
```

- AWS認証: GitHub Actions OIDC連携（IAMアクセスキー不使用）
- S3同期: `--delete` オプションで古いファイルを自動削除
- キャッシュ: CloudFrontキャッシュ全体無効化（`/*`）

※ インフラリソース（S3バケット、CloudFront、IAMロール等）は `cc_aws_portfolio` リポジトリのTerraformで管理

## スコープ外

明示的にスコープ外とする項目:
- サーバーサイドレンダリング（SSR）・API Routes — Static Export専用
- CMS連携（Contentful, microCMS等）— TSファイルでのデータ管理で十分
- コメント機能・問い合わせフォームのバックエンド — 静的サイトの範囲外
- インフラ構築（Terraform）— 別リポジトリ `cc_aws_portfolio` で管理
- ドメイン取得・DNS設定 — インフラ側の責務
